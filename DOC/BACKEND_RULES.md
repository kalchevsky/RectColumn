# Правила Backend

## Виртуальный датчик dT

- При `enabled == false` метод `poll()` обязан выставлять `present = true`, `value = NAN`, `error = false`, сбрасывать runtime тревог и управления и завершаться без расчёта `dT`.
- При `enabled == true` метод `poll()` обязан считать `dT` только если оба источника валидны: `present == true`, `enabled == true`, `error == false`, `value != NAN`.
- Если хотя бы один источник `dT` невалиден, метод `poll()` обязан выставлять `value = NAN` и `error = true`.
- Поля `_lastPollMs`, `hwLimited` и `diagCode` обязаны обновляться при каждом вызове `poll()` независимо от состояния `enabled`.

## Реле CH1..CH3 и подтверждение WER

- Таймауты подтверждения каналов задаются раздельно через `RELAY_CONFIRM_TIMEOUT_CH1_MS`, `RELAY_CONFIRM_TIMEOUT_CH2_MS`, `RELAY_CONFIRM_TIMEOUT_CH3_MS` в `config.h`.
- Значения таймаутов обязаны отражать физическое поведение конкретного исполнительного устройства и не выравниваются без аппаратного основания.
- Для текущей аппаратной конфигурации быстрый `CH1` (силовое реле 220 В) и более медленные `CH2`/`CH3` (клапаны) допускают разные таймауты подтверждения; выравнивать их «для красоты» нельзя.
- Если значение любого `RELAY_CONFIRM_TIMEOUT_CHx_MS` меняется, соответствующее правило в `DOC\BACKEND_RULES.md` обновляется в том же патче.

## API состояния ручного управления реле

- Для `CH1`, `CH2`, `CH3` состояние «команда принята, подтверждение ещё не пришло» отражается через `relayPending == true` и/или `pendingCmd != null` в ответах состояния.
- Подтверждённое состояние канала для UI ручного управления отражается через `feedbackOn` и `confirmActual`.
- Диагностический текст ошибки ручной команды, если он есть, передаётся через `relayErrorText` в готовом для показа виде.

## API причин квитирования

- `/api/v1/state` и `/api/v1/ack` могут дополнительно передавать массив `activeAlarmReasons` с уже сформированными текстами активных причин, требующих квитирования.
- Поле `activeAlarmReasons` использует существующие runtime-причины и не меняет семантику `activeAlarmCount`, `unackedAlarmCount` и самой логики квитирования.

## Ошибки и потеря датчиков (T1/T2/T3/P/F)

- Ошибка датчика равна потере датчика и является отдельным классом состояния, не совпадающим с превышением пользовательских уставок.
- Источники ошибки датчика ограничены аппаратной ошибкой чтения, выходом за физический sanity-range и устойчивым отсутствием валидного чтения дольше `SENSOR_LOST_TIMEOUT_MS` или `SENSOR_LOST_TIMEOUT_DS_MS`.
- При ошибке датчика он исключается из автоматики, а каналы, чьи правила используют этот датчик, переводятся в безопасное состояние «нет команды по этому датчику».
- Пока зафиксирована потеря датчика, тревоги превышения пользовательских уставок по этому датчику не выставляются; вместо них допускается ровно одно событие «Потеря датчика X».
- Датчик считается восстановленным только после непрерывных валидных показаний не короче `SENSOR_HEALTHY_HYSTERESIS_MS`; повторяющиеся строки `подключён/отключён` в журнале для одного и того же перехода запрещены.
- `sensorErrorLatched` снимается только циклом `OFF->ON` пользовательского флага `enabled` для соответствующего датчика; подтверждение тревог не снимает эту защёлку и не возвращает датчик в автоматику.
- Тексты уведомлений о потере датчиков ограничены формой `Потеря датчика T1`, `Потеря датчика T2`, `Потеря датчика T3`, `Потеря датчика P`, `Потеря датчика F`.
- Потеря датчика должна быть отражена в runtime и API не позже `SENSOR_LOST_TIMEOUT_MS` или `SENSOR_LOST_TIMEOUT_DS_MS` для соответствующего класса датчиков.
