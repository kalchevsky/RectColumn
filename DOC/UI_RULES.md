# Правила UI

## Карточки датчиков (страница конфигурации)

- В верхнем ряду карточки датчика допустимы только элементы `Статус: «Включено/Выключено»`, кнопка `Включить!/Отключить!`, поле `Период опроса` и кнопка `Установить другой` в их текущем порядке.
- Кнопок `Настроить управление` и `Настроить тревоги` в верхнем ряду карточки датчика нет ни у одного датчика, включая `dT`.
- Датчик `dT` рендерится тем же общим путём, что и температурные датчики `T1`, `T2`, `T3`; отдельной ветки рендера для `dT` нет.
- Настройки управления, завязанные на датчик, открываются из стандартного перехода `#/sensorCtrl/<id>` через общую сетку датчиков на стартовой странице.
- Настройки тревог, завязанных на датчик, открываются из стандартного перехода `#/sensorAlarm/<id>` через общую сетку датчиков на стартовой странице.

## Ручное управление реле (CH1..CH3)

- У кнопки `CH1`, `CH2`, `CH3` есть только три визуальных состояния: `off`, `on`, `pending`.
- Состояние `off` использует базовый синий стиль и подпись `Включить`.
- Состояние `on` использует активный стиль и подпись `Выключить`; оно соответствует подтверждённому включённому каналу по WER/feedback.
- Состояние `pending` использует отдельный нейтральный серый стиль, встроенный CSS-спиннер и атрибут `disabled`; оно допустимо только между принятием команды и финальным подтверждением/ошибкой.
- В состоянии `pending` кнопка обязана иметь `disabled` и `pointer-events: none`, чтобы повторные клики не проходили ни через DOM, ни через CSS-события.
- Визуальное состояние кнопки `CH1`, `CH2`, `CH3` определяется подтверждённым состоянием канала по WER/feedback и промежуточным состоянием ожидания подтверждения, а не фактом отправки команды само по себе.
- Состояние `pending` применяется только при явном признаке активной команды: `relayPending == true`, `pendingCmd != null` или локально запомненной активной команде до ближайшего ответа/опроса; отсутствие или `undefined` полей подтверждения не является основанием для `pending`.
- Состояние `pending` имеет приоритет над текущим `on` или `off`, если для канала уже принята новая команда и подтверждение по ней ещё не пришло.
- Состояние `pending` автоматически сменяется на `on` после подтверждения включения и на `off` после новой терминальной развязки для текущей команды; UI не должен залипать в `pending`.
- После клика `Включить`/`Выключить` по `CHx` состояние `pending` удерживается до терминальной развязки от прошивки; промежуточные «нейтральные» polling-ответы не снимают `pending` досрочно.
- Поведение `pending` одинаково для первого и для всех последующих нажатий в рамках одной сессии страницы; локальное состояние команды после терминальной развязки или ухода со страницы сбрасывается.
- В состоянии `pending` повторные клики по кнопке игнорируются.
- На стороне UI есть страховочный максимум удержания `pending`, близкий к максимальному `RELAY_CONFIRM_TIMEOUT_CHx_MS` с небольшим запасом; после его истечения кнопка возвращается в `off` и не остаётся в `pending` навсегда.
- Цвет состояния `pending` задаётся отдельными CSS-переменными `--btn-pending-bg`, `--btn-pending-fg`, `--btn-pending-spinner` и обязан быть заметно темнее фона страницы, отличаться и от базового синего `off`, и от активного `on`.
- Спиннер состояния `pending` реализуется встроенным CSS без внешних шрифтов и иконок.
- При первом рендере страницы и при отсутствии активной команды кнопка `CHx` отображается как `off`: базовый синий стиль, подпись `Включить`, без спиннера.
- Новые терминальные состояния `timeout`, `blocked`, `mismatch` и `faultLatched` для текущей команды имеют приоритет над `pending` и возвращают кнопку в `off`; унаследованные маркеры прошлого цикла, включая старые `relayError`/`relayErrorMs`, не должны досрочно снимать локально armed `pending`.
- При ошибке или таймауте подтверждения кнопка канала возвращается в исходный вид `Включить` с базовым синим стилем до получения подтверждения включения.
- Статусная строка под кнопкой показывает фактический текст ошибки от прошивки без переписывания на стороне UI, если прошивка передала `relayErrorText`.
- Поведение кнопок и статусных строк идентично для `CH1`, `CH2`, `CH3`.
- UI не отправляет автоматическую компенсирующую команду `Выключить` при ошибке подтверждения и не корректирует состояние прошивки сам.
- Один и тот же набор правил `off/on/pending` применяется и сразу после ручной команды, и после последующих обновлений состояния.
- Атрибуты `disabled` и `aria-busy` выставляются только тогда, когда `data-btn-state == "pending"`.
