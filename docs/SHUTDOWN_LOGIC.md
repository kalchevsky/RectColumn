# Логика остановки и отключения каналов

## Назначение документа

Этот документ фиксирует, как в проекте `RectColumn` сейчас разделены:

- тревоги датчиков (`alarm`);
- управляющие правила каналов (`control`);
- защитные отключения (`protection`);
- аварийные и операторские остановы (`safety`, `STOP`, manual).

Документ нужен как опорное описание для регрессионных тестов и для разбора спорных сценариев вида `F -> CH1`.

## 1. Слой `alarm`

За формирование обычных тревог отвечает `SensorBase::checkAlarms()`.

- Для аналоговых датчиков пороги берутся из `alarm[i]`.
- Для `alarmDelayMs` используется отдельный runtime `_alarmCandidateSinceMs[i]`.
- Пока нарушение не продержалось весь `alarmDelayMs`, `alarm[i].triggered` не поднимается.
- Если значение вернулось в норму раньше таймаута, таймер тревоги сбрасывается.

Важно:

- `alarmDelayMs` относится только к ветке тревог.
- Он не должен подменять и не заменяет `ctrlDelayMs`.

## 2. Слой `control`

За управление каналами по датчикам отвечает `OutputManager::_syncRuntimeState()` вместе с `SensorBase::evalCtrl()`.

### Как вычисляется команда

Каждое правило `sensor.ctrl[outIdx]` может вернуть:

- `1` — запрос на включение (`wantOn`);
- `-1` — запрет / автоотключение (`forbid`, `autoOff`);
- `0` — нейтральное состояние.

### Где применяется `ctrlDelayMs`

Задержка применения команд хранится в `SensorBase::ctrlDelayMs` и обрабатывается в `SensorBase::evalCtrl()`.

- Для каждого канала датчик хранит `candidate_cmd` и `candidate_since_ms`.
- Пока команда удерживается меньше `ctrlDelayMs`, `evalCtrl()` возвращает `0`.
- После истечения задержки возвращается реальная команда `1` или `-1`.
- Если состояние датчика стало нейтральным, runtime сбрасывается.

Следствие:

- до истечения `ctrlDelayMs` не должно быть `forbidden`;
- до истечения `ctrlDelayMs` не должно быть `autoOff`;
- до истечения `ctrlDelayMs` ручное включение не должно блокироваться через эту control-ветку.

## 3. Защитные правила `fixedOffOnly` для `L` и `F`

Для `L` и `F` на каналах `CH1..CH3` используются фиксированные OFF-only правила.

- В API они помечаются как `fixedOffOnly=true` и `purpose=protection`.
- Внутри прошивки такие правила нормализуются в:
  - `logic = LOGIC_COOL`
  - `minVal = 0.5`
  - `maxVal = 2.0`
- При `value=0` правило формирует именно `-1`, то есть защитное отключение.
- При `value=1` правило нейтрально и не требует включения канала.

После исправления `ctrlDelayMs` применяется к этим защитным правилам так же, как и к обычной control-ветке:

- `F=0` или `L=0` сначала запускает runtime-кандидат;
- канал отключается только после устойчивого нарушения в течение `ctrlDelayMs`;
- возврат значения в норму до таймаута обнуляет таймер.

## 4. Особенность схемы `F -> CH1`

В `OutputManager::_flowControlGate()` заложено отдельное правило схемы:

- защита `F -> CH1` считается актуальной, когда `CH2` уже включён или уже запрошен к включению;
- если `CH2` выключен и не запрашивается, потеря потока не должна запрещать `CH1`;
- для `CH2` и `CH3` защита по `F` действует только если именно их правила включены.

Из этого следуют два важных сценария:

- `F -> CH1` не должен выключать `CH2` и `CH3`, если их правила по `F` выключены;
- для `CH1` потеря потока должна отрабатываться только при активном или уже запрошенном `CH2`;
- для `CH2` и `CH3` отсчёт `ctrlDelayMs` должен стартовать уже от намерения держать канал включённым, а не только после прихода WER-подтверждения.

## 5. Когда runtime управления обязан сбрасываться

Runtime control-ветки нельзя переносить через изменение конфигурации.

Сейчас сброс делается в таких случаях:

- при `sensor/config`, если изменился `enabled`;
- при `sensor/config`, если изменился `ctrlDelayMs`;
- при `sensor/ctrl` для конкретного `sensor -> channel` правила;
- при переходе датчика в нейтральное состояние;
- при выключенном датчике;
- при отключённом правиле.

Важно:

- при кратком `controlGate=false` runtime больше не обнуляется, а только "замораживается";
- это нужно, чтобы краткий провал подтверждения или переходный процесс не запускал `ctrlDelayMs` заново;
- полноценный сброс runtime по-прежнему происходит только при нейтральной команде, выключении датчика, выключении правила или явном rearm после смены конфигурации.

Это защищает от ложного мгновенного отключения после:

- повторного включения датчика;
- повторного включения правила;
- изменения `logic`, `min`, `max`;
- переключения цифрового `fixedOffOnly` правила.

## 6. Слой `safety`

`ProcessSafety` отвечает за отдельную аварийную логику поверх обычного channel control.

### `L` и `F` как alarm-источник

Для `L` и `F` safety-слой использует `externalAlarmLogic`.

- Для `L` тревога поднимается как "Авария уровня".
- Для `F` тревога поднимается как "Авария потока".
- Для этих тревог используется именно `alarmDelayMs`, а не `ctrlDelayMs`.

Это отдельная ветка от channel-control:

- `alarmDelayMs` задерживает появление тревоги и alarm-событий;
- `ctrlDelayMs` задерживает именно защитное отключение канала по control-правилу.

### WER safety

WER-подтверждения живут в safety-слое как защёлкиваемая блокировка:

- при latched fault ставится `RULEIDX_SAFETY_WER`;
- канал принудительно выключается;
- блокировка держится до `POST /api/v1/safety/reset`.

## 7. `STOP` и manual

`STOP` работает отдельно от датчиков:

- блокирует основные каналы `CH1..CH3`;
- сбрасывает ручные удержания;
- не должен переиспользовать старые manual ON после release.

Ручное управление тоже отделено от тревог:

- manual ON отклоняется, если уже действует `forbidden` или `STOP`;
- manual OFF не должен ломать safety-логику;
- `CH4` и `CH5` не участвуют в WER-подтверждениях основных реле.

## 8. Диагностика `RELAY_OFF`

При отключении канала по ветке `channel_control` лог теперь должен содержать:

- `source=channel_control`;
- `sensorId=...`;
- `ctrlDelayMs=...`;
- `elapsedMs=...`;
- `ruleState=logic=...,min=...,max=...,runtimeCmd=...`;
- `sensorValue=...`.

Это нужно, чтобы различать:

- alarm-событие;
- safety-отключение;
- отключение именно по control/protection-правилу после выдержки `ctrlDelayMs`.
