- логика журнала событий (EventLog)

Не перетирай уже внесённые полезные изменения без необходимости.

Контекст: выявлены 4 бага. Их нужно разобрать, локализовать и исправить.

================================================================
БАГ 1. Датчик давления: при потере датчика нет ошибки, есть фантомное значение
================================================================

Симптомы:
- Датчик давления физически отключён (выдернут из разъёма).
- Система не показывает "ошибка датчика" сразу.
- Вместо ошибки отображается значение 1718,79 (фантомное значение из-за наводки/АЦП).
- Из-за этого срабатывает сигнализация и отключаются все каналы, по которым было включено управление.
- Уведомление "ошибка датчика" на экране всё-таки появилось, но через несколько минут.

Как должно быть:
- При потере датчика (ошибка) датчик перестаёт участвовать в управлении:
  * не включает ничего;
  * не выключает ничего;
  * не вызывает alarm/protection.
- Состояние "ошибка датчика" должно выставляться быстро, а не через несколько минут.
- Фантомное значение (например, 1718,79) не должно использоваться в логике управления.

Что сделать:
1. Найти, как сейчас определяется "ошибка датчика давления":
   - порог по значению (за пределами рабочего диапазона);
   - таймаут чтения;
   - флаг valid/invalid;
   - наличие связи с АЦП.
2. Ввести/уточнить диапазон валидных значений для датчика давления.
   - значения вне диапазона должны сразу помечаться как sensor error.
3. Сократить задержку перехода в состояние sensor error.
4. При sensor error датчик должен:
   - быть исключён из расчётов alarm/control/protection;
   - не порождать alarm по min/max;
   - не вызывать отключение каналов.
5. Логику "канал отключается из-за sensor error" пересмотреть:
   - если канал должен оставаться в текущем состоянии — оставить;
   - если канал должен переходить в безопасное состояние — описать это явно и логировать как sensor_lost, а не как alarm.

================================================================
БАГ 2. Датчики температуры: неверный текст уведомления и ложное "подключение"
================================================================

Симптомы:
- При отключении датчика температуры на экране появляется "ошибка датчика" — это правильно.
- Но уведомление приходит с текстом "T1min", хотя должно быть "потеря датчика T1".
- В журнал пишутся записи о попытках подключения датчика.
- В логе видно, что датчик якобы подключается, потом снова отключается, хотя физически он выдернут из разъёма.

Как должно быть:
- При потере датчика температуры уведомление должно явно говорить "потеря датчика T1" (и аналогично T2, T3...), а не "T1min".
- "T1min" должно приходить только при реальном падении температуры ниже минимума, а не при потере датчика.
- В журнале не должно быть ложных подключений датчика, если физически он отключён.
- Логика "датчик подключился" должна срабатывать только при реальной валидной температуре.

Что сделать:
1. Найти место, где формируется текст уведомления для датчика температуры:
   - проверить, не путается ли "ошибка" с "min".
   - при sensor error должен использоваться отдельный тип события: sensor_lost.
2. Уточнить, как сейчас принимается решение "датчик подключён":
   - возможно, любое валидное по диапазону значение считается подключением;
   - возможно, ложное значение проходит через фильтр.
3. Ввести/уточнить условие "датчик подключён":
   - стабильное валидное значение в течение N последовательных чтений;
   - значение в рабочем диапазоне;
   - отсутствие признаков обрыва/КЗ.
4. Уведомления и записи журнала по датчикам температуры должны различать:
   - sensor_lost;
   - sensor_restored;
   - T_min;
   - T_max.
5. Убедиться, что sensor_lost не превращается ошибочно в T_min или T_max.

================================================================
БАГ 3. После вставления датчика индикация ошибки исчезает сама
================================================================

Симптомы:
- При возврате датчика в разъём индикация "ошибка датчика" на экране пропадает автоматически.
- Это противоречит требованию: индикация "ошибка датчика" должна сниматься только пользователем — путём отключения и повторного включения датчика в интерфейсе.

Как должно быть:
- Если датчик ушёл в ошибку, индикация ошибки сохраняется до тех пор, пока пользователь:
  * не отключит датчик в интерфейсе;
  * и не включит его снова.
- Восстановление физического соединения не должно автоматически очищать состояние ошибки в UI.
- В журнал событий можно фиксировать факт восстановления связи отдельно (sensor_restored), но индикация ошибки в UI не снимается.

Что сделать:
1. Разделить две сущности:
   - фактическое текущее состояние датчика (live);
   - "залипающий" флаг ошибки (sticky error flag).
2. Sticky error flag должен:
   - выставляться при первой ошибке;
   - не сбрасываться при возврате валидного значения;
   - сбрасываться только при цикле "выключить датчик в UI -> включить датчик в UI".
3. В API/UI должно быть отдельное поле, по которому фронт показывает индикацию "ошибка датчика".
4. На фронте использовать именно sticky flag, а не live-состояние.
5. Если sticky flag активен, датчик не участвует в управлении, даже если live-значение уже валидное.
6. После сброса sticky flag пользователем (off -> on) логика возвращается к нормальной.

================================================================
БАГ 4. Реле 3 не включается; индикация и поведение кнопки некорректные
================================================================

Симптомы:
- Реле 3 не включается ни вручную, ни в автоматическом режиме.
- На стартовой странице индикация (точки) показывает правильное состояние — это корректно.
- На странице ручного управления индикация показывает "включено", хотя реле фактически не включилось.
- Есть сообщение, что подтверждения нет.
- После истечения таймаута подтверждения индикация не становится "выключено".
- При этом никакой команды на выключение подаваться не должно.
- Канал должен переходить в "нейтральное" состояние.
- Кнопка должна возвращаться в состояние, при котором можно только повторно отправить команду "включить".

Как должно быть:
- При команде "включить" реле 3:
  * подаётся импульс/команда включения;
  * ожидается подтверждение в течение таймаута;
  * если подтверждение пришло -> состояние "включено";
  * если подтверждения не пришло после таймаута:
    - НЕ подавать команду выключения;
    - перевести канал в нейтральное состояние;
    - индикация на странице ручного управления должна стать "выключено" / "нет подтверждения";
    - кнопка должна перейти в состояние, в котором доступна только повторная команда "включить".

Что сделать:
1. Проверить, что для реле 3 корректно настроены:
   - GPIO команды включения;
   - GPIO команды выключения (если применимо);
   - GPIO/сигнал подтверждения;
   - таймаут подтверждения;
   - логика "нейтральное состояние".
2. Найти, почему реле 3 не включается:
   - возможно, неверный пин;
   - возможно, инверсия сигнала;
   - возможно, конфликт с другим каналом;
   - возможно, защита/блокировка не даёт включить.
3. Исправить расхождение между индикациями:
   - стартовая страница;
   - страница ручного управления.
   Источник истины должен быть один — фактическое состояние канала, рассчитанное прошивкой.
4. После таймаута без подтверждения:
   - не подавать команду off;
   - перевести канал в нейтральное состояние;
   - выставить индикацию "выключено" / "нет подтверждения";
   - оставить кнопку в состоянии "можно повторить попытку включения".
5. Проверить логику ручного управления:
   - чтобы повторный клик не уходил в "выключить", если канал не подтвердился;
   - чтобы автоматический режим не пытался выключать канал, который и так не подтвердился.
