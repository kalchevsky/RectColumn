# RectColumn

Прошивка для **ESP32 DevKitC v4 (38-pin, WROOM-32)**, используемого в качестве контроллера.


# Возможные аппаратные ограничения !!!

- `PIN_C = GPIO39 = ADC1`. Это вход-only GPIO.
- `PIN_V` и `PIN_WER_CH3` используют общий GPIO35.
- `WER_CH1` на `GPIO27` использует внутреннюю подтяжку к `GND` (`INPUT_PULLDOWN`).
- `WER_CH2/CH3/CH4` на `GPIO34/35/36` остаются вход-only и требуют внешней подтяжки к `GND`.

# Примечание по werBell / buzzer

В текущем профиле проекта отдельный `werBell` **не используется**.

Причина:
- `CH5` уже назначен на `GPIO13` как buzzer output
- один и тот же GPIO нельзя корректно использовать одновременно как вход подтверждения и как выход

Если нужна поддержка `werBell`, это должно быть сделано:
1. либо в новой ревизии hardware-map с отдельным GPIO,
2. либо отдельным compile-time профилем.

# Логика автоматики CH1..CH3

Реле `CH1..CH3` работают по state-hold схеме:

1. сначала собирается сумма OFF/forbid по датчикам;
2. если `forbidMask != 0`, реле выключается;
3. иначе, если `wantOnMask != 0`, реле включается;
4. иначе применяется новая ручная команда;
5. если новой команды нет, реле сохраняет текущее состояние.

Это означает, что нейтральная зона между `minVal` и `maxVal` не выключает реле сама по себе. Реле удерживает последнее состояние до нового условия ON/OFF или новой команды оператора.

`CH4` и `CH5` не относятся к этой схеме. Они остаются level-driven, чтобы звонок и buzzer выключались при исчезновении причины звука.

## Датчики L/F

`L` и `F` для `CH1..CH3` являются фиксированными бинарными OFF-only условиями:

- `logic = LOGIC_COOL`;
- `minVal = 0.5`;
- `maxVal = 2.0`;
- `enabled` сохраняется пользовательским;
- `value = 0` после `ctrlDelayMs` формирует forbid/OFF;
- `value = 1` формирует нейтраль;
- ON-команду эти датчики не формируют.

Режим канала `HEAT/COOL` применяется только к датчикам схемы `T1/T2/T3/P`. Датчики `dT/C/V` оставлены как явное расширение: их индивидуальная логика не переписывается при смене режима канала.

## ProcessSafety

`ProcessSafety` оставлен как отдельный аварийный слой выше обычных sensor ctrl правил. Его запреты добавляются в `forbidMask` отдельными битами:

- `RULEIDX_SAFETY_LEVEL`;
- `RULEIDX_SAFETY_FLOW`;
- `RULEIDX_SAFETY_PRESSURE`;
- `RULEIDX_SAFETY_WER`.

В `/api/v1/state` для выходов доступны `sensorForbidMask`, `safetyForbidMask`, `effectiveForbidMask` и `forbidReasons`, чтобы отличать обычные sensor ctrl причины от аварийного слоя.
