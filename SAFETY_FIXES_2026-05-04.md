# Исправления от 2026-05-04

## Что исправлено в логике

1. Исправлена модель подтверждений `WER_*` в `EMU_MODE`.
   - По умолчанию подтверждения теперь работают в режиме `авто` и повторяют фактическое состояние выхода.
   - Добавлены режимы эмуляции `auto`, `force_on`, `force_off` для `WER_CH1..WER_CH4`.
   - Убран ложный сценарий, когда в обычной эмуляции `CH1` уходил в таймаут только потому, что `WER_CH1` не был вручную поднят.

2. Усилена диагностика ручного управления реле.
   - Причина блокировки ручного `ON` теперь расшифровывается: `STOP`, датчик, авария уровня, авария потока, авария давления, авария `WER`.
   - В API для состояний/команд добавлены текстовые поля:
     - `forbidReasonText`
     - `relayErrorText`
     - `detailText`
     - `confirmFaultText`
     - `confirmEmuMode`, `confirmEmuModeText`

3. Улучшен журнал событий.
   - Сообщения о подключении/отключении датчиков переведены на русский.
   - Сообщения об ошибках датчиков и тревогах переведены на русский.
   - Сообщения о включении/выключении выходов переведены на русский.
   - Сообщения `STOP` стали явными:
     - при активации фиксируется блокировка ручного включения `CH1..CH3`
     - при safety-останове фиксируется, что `STOP` защёлкнут
   - Сообщения по `WER` стали подробными:
     - таймаут подтверждения
     - подтверждение осталось включённым
     - подтверждение было `HIGH` до команды `ON`
     - восстановление после таймаута

4. Улучшена память о последней ошибке ручной команды.
   - Контроллер теперь хранит не только тип ошибки (`blocked`/`timeout`), но и конкретную причину последнего отказа.
   - Это используется для человекочитаемого статуса канала и журнала.

## Что исправлено в API

1. `/api/v1/emu/set`
   - Добавлена поддержка:
     - `WER_CH1_mode..WER_CH4_mode`
     - `W1_mode..W4_mode`
   - Старые булевы поля `WER_CHx` и `Wx` оставлены для совместимости.

2. `/api/v1/relay/*/state` и `/api/v1/state`
   - Добавлены поля для полной расшифровки запретов и ошибок ручного управления.

3. Ответы на ручные команды реле
   - `userMessage` теперь сообщает точную причину отказа, а не только общий факт блокировки.

## Важная ремарка

Сверка с "прикреплённым файлом" выполнена по уже утверждённой схеме и требованиям из переписки, потому что само вложение в рабочем каталоге и ресурсах сессии недоступно. Логика приведена к безопасному варианту с приоритетом:

1. аварии и защёлкнутые faults
2. `STOP`
3. safety-forbid
4. sensor forbid
5. sensor want
6. ручная команда
7. удержание последнего безопасного состояния
